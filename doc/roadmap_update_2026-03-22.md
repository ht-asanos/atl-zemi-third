## 2. 次の実装ステップ

### 推奨: Phase 4-3 レシピプール定期自動更新

次にやるべきことは `4-3 レシピプール定期自動更新`。

理由:

- 既にある楽天 / YouTube / 管理画面運用を手動保守のままにしておくコストが高い
- 品質ゲートやクリーンアップを導入しても、定期実行がなければレシピ鮮度と DB 健全性が維持しにくい
- 買い物リスト、再生成、favorites などの実装価値は、母集団レシピが安定更新されて初めて最大化される
- 写真アップロードや体組成ログよりも、直近の利用価値と運用安定性への寄与が大きい

---

## 3. 4-3 の具体的な実装順

### 3-1. 実行単位を固定する
まず、定期実行ジョブの責務を明確に分ける。

候補:

- recipe refresh
- recipe backfill
- non-meal prune
- nutrition rebuild または ingredient rebuild

1ジョブに全部詰めるより、ジョブ種別ごとに結果を追跡できる形がよい。

### 3-2. 実行履歴を永続化する
`job_logs` 相当のテーブルを追加し、少なくとも以下を保存する。

- `job_name`
- `started_at`
- `finished_at`
- `status`
- `triggered_by` (`system` / `manual`)
- `summary_json`
- `error_message`

ここがないと、失敗時の原因追跡と再実行判断が属人的になる。

### 3-3. 実行基盤を決める
第一候補は GitHub Actions の schedule 実行。

理由:

- 既存 repo 中心の運用に寄せやすい
- 実行ログを GitHub 側でも追える
- 段階的に導入しやすい

Supabase Edge Functions / external cron は代替案だが、現時点では GitHub Actions の方が導入コストが低い。

### 3-4. 再試行と通知を入れる
最低限必要なのは以下。

- 失敗時の再試行
- 最終失敗時の通知
- 成功 / 失敗のサマリ保存

通知先は Slack が現実的。メールは後回しでよい。

### 3-5. 手動実行との整合を取る
既存の CLI / API 手動運用を消す必要はない。
定期実行と手動実行が同じサービス層を叩く構造に寄せるべき。

必要な方針:

- CLI だけにロジックを閉じ込めない
- API と cron の両方から同じサービス関数を呼ぶ
- dry-run と execute を明確に分ける

---

## 4. 4-3 完了条件

以下を満たしたら 4-3 完了扱いでよい。

- 週次または日次のレシピ更新ジョブが自動実行される
- 実行結果が DB に記録される
- 失敗時に通知される
- 同一ジョブの再実行判断に必要なログが残る
- 手動実行と定期実行でロジック分岐が最小化されている

---

## 5. その次の優先順位

4-3 完了後の順序は以下を推奨する。

1. 4-5 食事写真アップロード
2. 5-1 体重・体組成の記録 API + 進捗グラフ UI
3. 6-2 API レスポンスタイム監視

理由:

- 4-5 は日次ログ体験の改善に直結する
- 5-1 は継続利用の価値を高めるが、まず運用安定が先
- 6-2 は本来重要だが、現段階ではまずデータ供給系ジョブの自動化が先
