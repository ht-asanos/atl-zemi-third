# 一人暮らし向け自炊 x トレーニング最適化アプリ 要件定義・実装計画（v1.0）

## 1. 目的
- 一人暮らしユーザーが「自炊」と「トレーニング」を無理なく習慣化できる状態を作る。
- 目的（ダイエット / 筋力アップ / ボルダリング向け）に応じて、毎日の食事・運動メニューを自動生成し、実行後のフィードバックで次回提案を適応させる。

## 2. ターゲット
- 20-40代の一人暮らし
- 自炊は最小工数で済ませたい
- トレーニングを継続したいが、何をどれくらいやるべきか分からない
- ボルダリングのパフォーマンスを伸ばしたい

## 3. 成功指標（KPI）
- 週次実行率（食事/運動の完了チェック）: 70%以上
- 4週間継続率: 60%以上
- 提案再生成後の再実行率: 50%以上
- ボルダリング目的ユーザーの自己評価改善（握力/保持力/体幹）: 4週間で平均+1段階

## 4. 機能要件

### 4.1 初期設定
- 入力項目: 年齢、性別、身長、体重、活動レベル、目的、アレルギー、調理可能時間
- 目的: `diet` / `strength` / `bouldering`

### 4.2 栄養計算エンジン
- BMR: Mifflin-St Jeor
- TDEE: BMR x 活動係数
  - 低い: 1.2
  - やや低い: 1.375
  - 中程度: 1.55
  - 高い: 1.725
- 目的別カロリー目標
  - diet: TDEE - 400 kcal
  - strength: TDEE + 250 kcal
  - bouldering: TDEE + 0-150 kcal
- PFC制約
  - タンパク質: 1.6-2.2 g/kg
  - 脂質: 0.6-1.0 g/kg
  - 炭水化物: 残カロリー

### 4.3 食事提案ロジック（引き算）
- ユーザーが主食を選択（例: 冷凍うどん1玉）
- 主食の kcal/PFC を日次目標から減算
- 残枠に対し以下を優先して組み合わせ提案
  - タンパク源: 卵、ツナ、サラダチキン、豆腐
  - かさ増し/微量栄養素: きのこ、冷凍野菜、海藻
- 制約
  - 1食調理時間 <= 10分
  - 1食予算上限（MVP初期値） <= 500円

### 4.4 トレーニング提案ロジック
- 目的別テンプレートから日次メニューを生成
- `diet`
  - 全身の大筋群中心（脚/胸/背中）
  - 消費増を狙った中強度
- `strength`
  - 部位別高負荷（漸進性過負荷）
- `bouldering`
  - 引く力（背中/上腕二頭筋）
  - 保持力（前腕）
  - 体幹（腹圧・抗回旋）
  - 押す力（胸/三頭）は補助的に少量

### 4.5 感想解析と適応
- 入力: テキスト感想、完了チェック、食事写真（任意）
- NLP（LLM）でタグ抽出し、最終決定はルールベースで実施
- 主要タグと適応例
  - `too_hard`: 次回ボリューム -20%
  - `cannot_complete_reps`: レップ -20% または負荷段階を1つ下げる
  - `forearm_sore`: 翌日は前腕高負荷を除外し体幹/下半身へ置換
  - `bored_staple`: 同等カロリー主食を3候補提示
  - `too_much_food`: かさ増し食材 -30%

### 4.6 ユーザーフロー
1. 初期設定
2. 主食選択
3. 1週間メニュー生成（食事 + 運動）
4. 日々実行
5. 感想送信（テキスト/写真）
6. 即時フィードバック表示
7. 翌日メニューへ自動反映

## 5. 非機能要件
- API応答時間: P95 < 2秒（生成除く通常API）
- プラン再生成: P95 < 5秒
- 可用性: 99.5%（MVP目標）
- セキュリティ
  - 画像バケットは private
  - 行レベルアクセス制御（ユーザー自身のデータのみ）
  - 退会時データ削除に対応

## 6. 技術構成
- Frontend: Next.js（App Router, TypeScript）
- Backend: FastAPI（Python）
- DB/Auth/Storage: Supabase（PostgreSQL + Auth + Storage）
- AI: ChatGPT API（感想のタグ抽出）

## 7. データ要件（MVP）
- `profiles`: 基本情報、活動係数、目的
- `goals`: カロリー/PFC目標
- `food_master`: 食材マスタ（kcal/PFC/価格/調理時間）
- `daily_plans`: 日次提案（食事/運動）
- `meal_logs`: 食事実績、写真URL、満足度
- `workout_logs`: 種目、セット、レップ、RPE
- `feedback_tags`: NLP抽出タグ
- `plan_revisions`: 再生成差分と理由

## 8. MVPスコープ
- 対応目的: 3種（diet/strength/bouldering）
- 主食カテゴリ: 5種（うどん、米、オートミール、パスタ、パン）
- 感想適応: まずは5タグのみ
- 週次メニュー生成 + 日次微調整

## 9. 受け入れ基準
- ユーザーが初期設定完了後、3目的すべてで1週間分メニューが生成される
- 主食選択変更で食事提案が再計算される
- `forearm_sore` 入力時、翌日の前腕高負荷種目が除外される
- `bored_staple` 入力時、主食代替案が3件提示される
- ユーザーごとのデータ分離が確認できる

## 10. 実装フェーズ
1. Phase 1: データモデルと計算エンジン
2. Phase 2: 週次プラン生成API + UI
3. Phase 3: ログ収集 + 感想解析 + 適応
4. Phase 4: 写真アップロード、運用監視、微調整

## 11. リスクと対策
- LLM出力の揺らぎ
  - 対策: タグ抽出のみに限定、意思決定はルール化
- 入力負荷で継続率が落ちる
  - 対策: 1タップ完了チェック、感想テンプレート
- ボルダリング向け負荷過多
  - 対策: 前腕疲労タグの優先度を最上位に設定
